# SeoulMySoulMate 프로젝트 완성 프롬프트

아래 프롬프트는 현재 저장소를 기준으로 남은 기능을 점검하고 프로젝트를 완성할 때 사용한다.

---

너는 `SeoulMySoulMate` 프로젝트를 마무리하는 풀스택 개발자야. 이 프로젝트는 서울 관광정보를 한곳에서 찾아보고, 장소별 후기를 공유하고, 챗봇 추천을 여행 일정에 바로 추가할 수 있는 서비스야.

단순히 새 코드를 많이 만드는 것이 아니라, 현재 저장소에 구현된 기능을 먼저 확인하고 기획 의도와 맞지 않거나 빠진 부분을 보완해 줘. 분석만 하고 끝내지 말고 실제 코드 수정, 실행 확인, 문서 정리까지 진행해 줘.

## 프로젝트 목표

서울 여행을 준비하는 관광객이나 주말 나들이 장소를 찾는 사용자가 다음 흐름을 자연스럽게 이용할 수 있어야 해.

1. 서울 관광 콘텐츠를 카테고리나 키워드로 찾는다.
2. 장소의 상세 정보와 다른 사용자의 후기 및 실시간 정보를 확인한다.
3. Soul Chat에서 원하는 지역, 관심 분야, 여행 시간에 맞는 장소를 추천받는다.
4. 콘텐츠 목록, 상세 화면, 지도, 챗봇 추천 카드에서 장소를 여행 일정에 추가한다.
5. 장소와 연결된 커뮤니티 글을 작성하고 댓글과 좋아요로 정보를 나눈다.

## 기술 스택과 기본 구조

- 프론트엔드: Vue 3, Vite, Vue Router, Pinia
- 백엔드: FastAPI, SQLAlchemy
- 데이터베이스: SQLite
- 원본 데이터: `data/서울_*.json` 7종
- 백엔드 경로: `backend/`
- 프론트엔드 경로: `frontend/`
- 데이터 스키마와 출처: `data/SCHEMA.md`, `data/SOURCE.md`
- 상세 기능 기준: `FEATURE_REQUIREMENTS.md`

기술 스택은 특별한 이유가 없으면 바꾸지 말고, 이미 사용 중인 구조와 코드 스타일을 유지해 줘. 새로운 라이브러리는 기존 코드만으로 해결하기 어려울 때만 추가해 줘.

## 먼저 확인할 내용

작업을 시작하기 전에 다음 내용을 확인해 줘.

- `git status`로 기존 변경 사항을 확인하고 사용자가 작업한 내용은 덮어쓰지 않는다.
- `README.md`, `FEATURE_REQUIREMENTS.md`, 기획서와 현재 코드를 함께 비교한다.
- 백엔드 모델, 스키마, 라우터와 프론트엔드 API 호출 형식이 서로 맞는지 확인한다.
- 화면이 존재하는지만 보지 말고 실제 API 연결, 저장, 새로고침 후 복원, 오류 처리까지 확인한다.
- 기획서의 초기 MVP와 현재 구현이 다르면 현재 구현된 인증, 지도, 테마, 실시간 정보 기능을 유지한다. 대신 핵심 사용자 흐름이 깨지지 않도록 정리한다.
- 하드코딩된 임시 데이터나 동작하지 않는 버튼, 사용하지 않는 코드가 있는지 확인한다.

## 필수 기능

### 1. 관광 데이터 적재와 콘텐츠 탐색

- 서버가 처음 실행될 때 `data/서울_*.json` 7종을 SQLite에 적재한다.
- 요청할 때마다 JSON 전체를 읽지 않고 SQLite에서 조회한다.
- 관광지, 레포츠, 문화시설, 쇼핑, 숙박, 여행코스, 축제공연행사를 한 서비스에서 제공한다.
- 콘텐츠 목록에서 카테고리, 주소 또는 지역, 장소명 키워드 검색과 페이지네이션이 동작해야 한다.
- 상세 화면에는 장소명, 카테고리, 주소, 연락처, 대표 이미지, 지도 위치 등 제공 가능한 정보를 표시한다.
- 이미지나 주소, 좌표가 없는 데이터도 화면이 깨지지 않게 처리한다.
- 원본 JSON은 수정하지 않고 데이터 출처와 라이선스 표기를 유지한다.

### 2. 회원 인증과 권한 처리

- 이메일, 닉네임, 비밀번호로 회원가입과 로그인이 가능해야 한다.
- 비밀번호와 세션 토큰 원문을 데이터베이스에 저장하지 않는다.
- 로그인 상태는 새로고침 후에도 복원되고, 만료되거나 잘못된 토큰은 안전하게 제거한다.
- 게시글과 댓글의 수정 및 삭제는 작성자만 할 수 있어야 한다.
- 기존 익명 게시글처럼 작성자 계정이 없는 데이터도 조회할 수 있게 호환성을 유지한다.
- 인증이 필요한 화면에 비로그인 사용자가 접근하면 로그인 후 원래 화면으로 돌아갈 수 있게 한다.

### 3. 장소 중심 커뮤니티

- 새 게시글은 관광 콘텐츠 한 곳을 선택해 연결한다.
- 콘텐츠 상세 화면에서 글쓰기로 이동하면 해당 장소가 자동으로 선택된다.
- 커뮤니티 목록에는 장소 이미지, 장소명, 카테고리, 주소, 글 제목과 내용 요약을 함께 표시한다.
- `content_id`, `category`, `keyword` 조건과 최신순, 좋아요순, 댓글순 정렬이 동작해야 한다.
- 후기가 많은 장소를 인기 장소 영역에 표시하고 선택 시 해당 장소의 글만 조회한다.
- 게시글 작성, 조회, 수정, 삭제와 댓글 작성 및 삭제, 좋아요 추가 및 취소가 실제 API와 연결되어야 한다.
- 로딩, 빈 결과, 권한 없음, 삭제 확인, API 실패 상태를 사용자에게 알아보기 쉽게 표시한다.

### 4. SQLite 근거형 Soul Chat

- 사용자의 문장에서 서울 지역, 관심 카테고리, 여행 시간이나 추천 개수를 파악한다.
- 챗봇이 보여 주는 답변과 추천 카드는 SQLite에서 검색된 관광 콘텐츠만 근거로 사용한다.
- OpenAI API를 사용할 때는 전체 원본 데이터를 보내지 않고 검색된 상위 결과와 필요한 대화 기록만 전달한다.
- `backend/.env`에 API 키가 있으면 OpenAI 모드, 없으면 SQLite 검색 기반 답변으로 동작한다.
- API 키를 추가한 뒤 서버를 다시 시작하지 않아도 다음 요청부터 반영되게 한다.
- 대화 내역과 이전 추천 문맥을 유지하고, 추천 카드에서 상세 보기와 일정 추가가 가능해야 한다.
- 연결 상태와 현재 모드를 화면에서 확인할 수 있게 한다.
- 검색 결과가 없거나 외부 API 요청이 실패해도 기본 검색 답변으로 안전하게 처리한다.

### 5. 여행코스 플래너

- 콘텐츠 목록, 상세 화면, 지도, 챗봇 추천 카드에서 장소를 일정에 추가할 수 있어야 한다.
- 날짜별로 장소를 구분하고 방문 시간, 메모, 순서를 수정할 수 있어야 한다.
- 일정 항목의 추가, 수정, 순서 변경, 삭제가 모두 동작해야 한다.
- 로그인하지 않아도 브라우저별 식별값을 이용해 새로고침 후 일정을 복원한다.
- 같은 장소를 중복 추가하거나 날짜가 비정상적인 경우를 적절히 처리한다.
- 이번 범위에서는 실시간 교통을 이용한 최적 경로 계산이나 내비게이션 기능까지 만들 필요는 없다.

### 6. 전체 장소 지도

- `/map`에서 좌표가 있는 관광 콘텐츠 전체를 탐색할 수 있어야 한다.
- 많은 마커는 클러스터링하고 카테고리, 주소, 장소명으로 필터링한다.
- 지도 마커나 검색 결과를 선택하면 콘텐츠 상세 화면으로 이동할 수 있어야 한다.
- 카카오맵 키가 없거나 SDK 로딩에 실패했을 때 원인과 설정 방법을 안내하고 다른 기능은 계속 사용할 수 있게 한다.
- 서울 자치구 경계와 혼잡도 표시가 현재 기능과 자연스럽게 연결되는지 확인한다.

### 7. 실시간 서울 정보

- 장소와 서울시 실시간 도시데이터 관측 지역을 연결해 혼잡도, 인구, 따릉이 등 제공 가능한 정보를 보여 준다.
- API 키가 없거나 데이터 제공처가 응답하지 않을 때 페이지 전체가 실패하지 않게 한다.
- 캐시와 재시도 처리를 사용해 같은 외부 요청을 불필요하게 반복하지 않는다.
- 실시간 정보와 관광공사 원본 정보의 출처를 화면에서 구분한다.

### 8. 화면 완성도와 사용성

- 웰컴, 콘텐츠 목록·상세, 지도, 로그인·회원가입, 커뮤니티, 챗봇, 플래너 화면의 이동 흐름을 확인한다.
- 모바일과 데스크톱에서 주요 기능을 사용할 수 있도록 반응형으로 구성한다.
- 라이트·다크 모드에서 글자, 카드, 입력창, 지도 주변 UI가 모두 읽기 쉬워야 한다.
- 사용자가 선택한 테마를 저장하고 첫 방문에는 운영체제 설정을 따른다.
- 모든 주요 화면에 로딩, 빈 상태, 오류, 재시도 UI를 제공한다.
- 버튼과 입력창의 키보드 접근, 포커스 표시, 대체 텍스트, 색상 대비를 기본 수준 이상으로 맞춘다.
- 브라우저 콘솔에 불필요한 오류나 경고가 남지 않게 한다.

## API와 보안 기준

- 현재 REST API 경로와 요청·응답 형식을 먼저 확인하고 프론트엔드와 일치시킨다.
- 최소한 콘텐츠, 인증, 게시글·댓글·좋아요, 챗봇, 플래너, 지도, 실시간 정보 API의 정상·실패 응답을 확인한다.
- 입력값 길이와 형식을 백엔드에서 검증하고, 존재하지 않는 데이터에는 알맞은 HTTP 상태 코드를 반환한다.
- `OPENAI_API_KEY`, 서울시 API 키 같은 비밀 값은 `backend/.env`에서만 관리한다.
- 카카오 JavaScript 키 외의 서버용 비밀 값을 `VITE_` 환경변수에 넣지 않는다.
- 실제 키나 토큰을 코드, 문서, Git 기록에 넣지 않는다.
- 필요한 항목만 들어 있는 `backend/.env.example`과 `frontend/.env.example`을 제공한다.
- CORS는 개발 환경에서 필요한 출처만 허용하고, 오류 메시지에 비밀 값이나 내부 정보가 노출되지 않게 한다.

## 작업 순서

1. 현재 구현 기능과 미완성·오류 항목을 짧게 정리하고 우선순위를 정한다.
2. 데이터 적재와 백엔드 API 계약을 먼저 안정화한다.
3. 프론트엔드의 실제 API 연결과 핵심 사용자 흐름을 완성한다.
4. 인증, 권한, 예외 상황, 반응형 UI를 점검한다.
5. 백엔드 테스트와 프론트엔드 빌드를 실행하고 발견된 문제를 수정한다.
6. 실행 방법, 환경변수, 데이터 출처, 주요 기능을 `README.md`에 최신 상태로 반영한다.

한 번에 전체 구조를 다시 만들지 말고, 확인한 문제를 기준으로 필요한 부분만 수정해 줘. 중간에 합리적으로 판단할 수 있는 내용은 기존 코드와 기획 의도를 기준으로 진행하고, 결과가 크게 달라지는 선택이 필요한 경우에만 질문해 줘.

## 완료 기준

다음 항목을 모두 확인한 뒤 작업을 마무리해 줘.

- 백엔드 서버가 오류 없이 실행되고 `/api/health`가 정상 응답한다.
- JSON 7종이 새 데이터 경로에서 적재되고 콘텐츠 검색과 상세 조회가 동작한다.
- 회원가입, 로그인, 로그아웃, 작성자 권한 처리가 동작한다.
- 장소 검색부터 상세 조회, 커뮤니티 글 작성, 챗봇 추천, 일정 추가까지 핵심 흐름이 이어진다.
- 지도, 테마, 실시간 정보가 설정 유무와 실패 상황에서도 화면 전체를 깨뜨리지 않는다.
- 새로고침 후 로그인 상태, 채팅 문맥, 테마, 일정이 각 기능의 저장 방식에 맞게 유지된다.
- 프론트엔드 `npm run build`가 통과한다.
- 가능한 범위에서 백엔드 자동 테스트를 실행하고, 테스트가 없다면 핵심 API를 직접 검증한다.
- `.env`, API 키, 세션 토큰, 로컬 DB 같은 민감하거나 실행 중 생성되는 파일이 Git에 포함되지 않는다.
- 변경한 파일, 구현한 기능, 실행한 테스트, 남아 있는 제한 사항을 마지막에 간단히 정리한다.
